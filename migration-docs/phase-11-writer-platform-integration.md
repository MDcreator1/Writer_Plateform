# Phase 11 - Writer Platform / Writing Studio Integration

Date: 2026-06-23
Status: Implemented; awaiting approval

## Outcome

Writer Platform and Writing Studio remain separate applications with separate responsibilities.

- Writer Platform owns story metadata, covers, slug, genres, tags, summary, SEO, pricing, publication, visibility, and analytics.
- Writing Studio owns chapters, drafts, draft promotion, Naming Board, Facts Board, notes, revisions, and editor state.
- Studio has two storage modes only: local folder and cloud story.
- The previous revision/status/re-sync subsystem has been removed. Cloud mode now behaves like the local folder model, with the same files stored in PostgreSQL.

## Current architecture

### Writer Platform

The Platform is a Next.js application backed by PostgreSQL and Prisma. `Story` remains the publishing aggregate and metadata authority. Reader-facing `Chapter` rows remain a projection used by the website and purchase/reading flows.

Creating a story in Admin creates a unique Story ID and a cloud Studio workspace. The story can be published or unpublished; both states are editable in Writing Studio.

### Writing Studio

Writing Studio remains a standalone application under `Writer_studio/` and is served by its own Node process. It has not been moved into the website.

Local projects continue using File System Access API/IndexedDB/local fallback storage. Cloud projects use the same virtual file-handle interface, but file reads and writes go to the Platform database API.

## Required changes and implementation

### Platform-created stories

- A unique `Story.id` is generated by the Platform.
- Metadata is stored only in `Story` and related Platform fields.
- A one-to-one `StudioProjectLink` and canonical workspace are created/materialized.
- Studio receives story identity and title for catalog/display purposes.
- Studio cannot write cover, slug, summary, genres, tags, SEO, pricing, visibility, or publication metadata.
- New stories receive the standard Studio folders/files plus an initial draft file, so they open like a newly created local project.

### Studio-only projects

- Remain local until the author explicitly chooses a cloud action.
- Do not create `Story`, `StudioProjectLink`, folder, or file rows.
- Do not appear on Writer Platform.
- Preserve the existing local-first workflow.

### Saving or publishing a local Studio project

The Platform panel offers:

- Save as Cloud Draft
- Publish Project

Both actions send the complete Studio workspace file set. Story creation and project linking occur in one transaction and are idempotent by `projectId`.

After conversion, Platform owns metadata while Studio continues to own authoring files.

### Published and unpublished cloud stories

Both use the same cloud workspace model. Publication state does not change the authoring storage format. Studio can create/edit/promote chapters and drafts in either state.

Chapter publication intent is stored in the managed Studio manifest. Updating a cloud authoring file directly refreshes the reader-facing `Chapter` projection; there is no separate manual sync command, revision counter, or re-sync queue.

## Database changes

### `Story`

Publishing metadata and settings remain authoritative here. `published`/`publicationStatus` distinguish published and unpublished stories without changing their Studio workspace format.

### `StudioProjectLink`

The one-to-one bridge contains only durable identity/storage information:

- unique `projectId`
- unique `storyId`
- project title and source
- workspace format version
- workspace materialization timestamp
- folder/file relations

Removed fields include sync status, revisions, hashes/snapshots, imported/pending counters, sync errors, and re-sync timestamps.

### `StudioProjectFolder`

Stores canonical project-relative folders with a unique `(studioProjectId, path)` constraint.

Canonical folders:

- `Chapters/`
- `Drafts/`
- `Trash/`
- `Edited_Chapter/`

### `StudioProjectFile`

Stores exact project-relative files, kind, JSON/text format, publication/authoring state, content hash, and timestamps. JSON uses JSONB. Text uses AES-256-GCM encryption.

Canonical files include:

- `Chapters_info.json`
- `Story_Naming.json`
- `Story_Drafts.json`
- `Trash/Trash_Drafts.json`
- `Temp_Chapter_Draft.json`
- `Chapters/chapter_XX.txt`
- `Drafts/draft_XX.txt`
- `Trash/draft_XX.txt`
- `Edited_Chapter/chapter_edit_XX.txt`

### `Chapter`

`studioDocumentId` provides stable identity between a Studio document and its reader-facing projection. The obsolete `sourceRevision` column has been removed.

### Removed database structures

- `StudioDraft`
- `StudioSyncAttempt`
- `StudioSyncStatus`
- `StudioDraftKind`
- all sync/re-sync fields on `StudioProjectLink`
- `sourceRevision` on Studio files and chapters

`prisma/migrations/20260623143000_remove_studio_sync_system/migration.sql`

## API changes

| Endpoint | Responsibility |
| --- | --- |
| `GET /api/studio/projects` | List cloud stories with Story ID, title, publication state, linked project ID, file count, and update time |
| `POST /api/studio/projects` | Idempotently ensure a Platform story has one cloud Studio project |
| `POST /api/studio/projects/create-draft` | Create/link an unpublished Platform story from a local project and store its full workspace |
| `POST /api/studio/publish` | Create/link a published Platform story from the Publishing Wizard and store its full workspace |
| `GET/PUT/DELETE /api/studio/projects/[projectId]/files` | List/read/write/delete canonical cloud workspace files |

Removed endpoints:

- `/api/studio/projects/[projectId]/sync`
- `/api/studio/projects/[projectId]/status`
- `/api/admin/studio-projects/[linkId]/resync`

Studio API authorization remains limited to an Admin session, configured integration key, or trusted development Studio origin.

## Cloud storage design

`VirtualDirectoryHandle` and `VirtualFileHandle` allow the existing Studio editor to use the same project-file operations in local and cloud modes.

- Local mode reads/writes the selected folder and local caches.
- Cloud mode reads/writes `StudioProjectFile` records through the file API.
- The managed manifest is sanitized server-side before storage.
- Platform title, Story ID, project link, source, and publication identity always win.
- Naming, facts, drafts, notes, revision drafts, editor formatting, and chapter text remain Studio-owned.
- Cloud writes update the reader-facing chapter projection transactionally after file persistence.

## Conflict resolution strategy

### Metadata

Writer Platform always wins. Studio APIs do not accept Platform metadata during normal authoring-file writes. Managed Studio metadata controls remain read-only.

### Duplicate stories/projects

- `StudioProjectLink.projectId` is unique.
- `StudioProjectLink.storyId` is unique.
- Draft/publish routes check an existing project link before story creation.
- Story and link creation are transactional.
- Slugs remain database-unique.

### File writes

- `(studioProjectId, path)` is unique.
- Paths reject traversal, empty paths, and excessive length.
- Content hashes make repeated writes idempotent.
- Server-side manifest sanitization prevents metadata ownership conflicts.
- Stable `studioDocumentId` values prevent chapter-title collisions.

### Offline/local-first behavior

Local-only work remains independent of Platform availability. Cloud projects keep the Studio virtual-file cache for immediate editor continuity, while PostgreSQL is the authoritative cloud store.

## Chapter status refinement

Chapter status now follows Writing Studio storage collections directly:

- `PUBLISHED`: a draft has been promoted/published into the Studio Chapters collection. It is reader-visible when its parent story is published.
- `DRAFT`: the document remains in `Story_Drafts.json` and `Drafts/`; it is not reader-visible.
- `TRASH`: the document remains in `Trash/Trash_Drafts.json` and `Trash/`; it is in the Studio recycle bin and is not reader-visible.

`ChapterStatus.ARCHIVED` was renamed to `TRASH` by migration `prisma/migrations/20260623180000_refine_chapter_statuses/migration.sql`.

Document identity is preserved while moving between collections. Draft promotion now changes the same document from `DRAFT` to `PUBLISHED`; moving a published chapter back to drafts changes that same document to `DRAFT`. Deleting a draft moves the same document to `TRASH`, and restoring it returns it to `DRAFT`.

Writing Studio's former **Save as Chapter** action is now **Publish Chapter** with the same icon and interaction. The Platform publication toggles were removed because promoted chapters are published by definition.

The Admin story details page now renders three separate classified sections—Published Chapters, Draft Chapters, and Trash—with independent counts, status colors, and deep links that open the matching Studio collection.
## Reader rendering refinement

Writer Portfolio's protected reader now uses the same content interpretation as Writing Studio review mode.

- Studio HTML is converted into readable review text before it is encrypted into the reader-facing `Chapter` projection.
- Paragraph tags, line breaks, block boundaries, Studio paragraph-gap markers, and HTML entities are normalized.
- Script/style/embed content and all remaining markup are discarded rather than injected into the page.
- The Reader applies the same parser again for legacy/stale payloads and renders React text blocks; it does not use `dangerouslySetInnerHTML`.
- Extra blank-line spacing from Studio is retained as controlled paragraph spacing.
- Existing cloud stories were backfilled so previously stored literal `<p>`/`<br>` syntax no longer appears.

Validation result: 32 published chapter projections scanned, 0 literal HTML-tag failures.
